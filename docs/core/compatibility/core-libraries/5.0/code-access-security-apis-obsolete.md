<span data-ttu-id="94900-104">.NET의 CAS(코드 액세스 보안) 관련 형식은 대부분 경고로 표시되며 사용되지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-104">Most code access security (CAS)-related types in .NET are now obsolete as warning.</span></span> <span data-ttu-id="94900-105">여기에는 <xref:System.Security.Permissions.SecurityPermissionAttribute>, CAS 권한 개체(예: <xref:System.Net.SocketPermission>), <xref:System.Security.Policy.EvidenceBase> 파생 형식 및 기타 지원되는 API와 같은 CAX 특성이 포함됩니다.</span><span class="sxs-lookup"><span data-stu-id="94900-105">This includes CAS attributes, such as <xref:System.Security.Permissions.SecurityPermissionAttribute>, CAS permission objects, such as <xref:System.Net.SocketPermission>, <xref:System.Security.Policy.EvidenceBase>-derived types, and other supporting APIs.</span></span>

## <a name="change-description"></a><span data-ttu-id="94900-106">변경 내용 설명</span><span class="sxs-lookup"><span data-stu-id="94900-106">Change description</span></span>

<span data-ttu-id="94900-107">.NET Framework 2.x - 4.x에서 CAS 특성 및 API는 CAS 요청 스택이 성공하거나 실패하도록 하는 등 코드 실행 과정에 영향을 줄 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-107">In .NET Framework 2.x - 4.x, CAS attributes and APIs can influence the course of code execution, including ensuring that CAS-demand stack walks succeed or fail.</span></span>

```csharp
// In .NET Framework, the attribute causes CAS stack walks
// to terminate successfully when this permission is demanded.
[SocketPermission(SecurityAction.Assert, Host = "contoso.com", Port = "443")]
public void DoSomething()
{
    // open a socket to contoso.com:443
}
```

<span data-ttu-id="94900-108">.NET Core 2.x - 3.x에서 런타임은 CAS 특성이나 CAS API를 적용하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-108">In .NET Core 2.x - 3.x, the runtime does not honor CAS attributes or CAS APIs.</span></span> <span data-ttu-id="94900-109">런타임은 메서드 항목의 특성을 무시하고 대부분의 프로그래밍 API는 영향을 주지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-109">The runtime ignores attributes on method entry, and most programmatic APIs have no effect.</span></span>

```csharp
// The .NET Core runtime ignores the following attribute.
[SocketPermission(SecurityAction.Assert, Host = "contoso.com", Port = "443")]
public void DoSomething()
{
    // open a socket to contoso.com:443
}
```

<span data-ttu-id="94900-110">또한 광범위한 API에 대한 프로그래밍 방식 호출(`Assert`)은 항상 성공하지만 제한적 API에 대한 프로그래밍 방식 호출(`Deny`, `PermitOnly`)은 런타임에 항상 예외를 throw합니다.</span><span class="sxs-lookup"><span data-stu-id="94900-110">Additionally, programmatic calls to expansive APIs (`Assert`) always succeed, while programmatic calls to restrictive APIs (`Deny`, `PermitOnly`) always throw an exception at run time.</span></span> <span data-ttu-id="94900-111">(<xref:System.Security.Permissions.PrincipalPermission>은 이 규칙의 예외입니다.</span><span class="sxs-lookup"><span data-stu-id="94900-111">(<xref:System.Security.Permissions.PrincipalPermission> is an exception to this rule.</span></span> <span data-ttu-id="94900-112">아래의 [권장 사항](#cas-action) 섹션을 참조하세요.)</span><span class="sxs-lookup"><span data-stu-id="94900-112">See the [Recommended action](#cas-action) section below.)</span></span>

```csharp
public void DoAssert()
{
    // The line below has no effect at run time.
    new SocketPermission(PermissionState.Unrestricted).Assert();
}

public void DoDeny()
{
    // The line below throws PlatformNotSupportedException at run time.
    new SocketPermission(PermissionState.Unrestricted).Deny();
}
```

<span data-ttu-id="94900-113">.NET 5 이상 버전에서 대부분의 CAS 관련 API는 더 이상 사용되지 않으며 `SYSLIB0003` 컴파일 시간 경고를 생성합니다.</span><span class="sxs-lookup"><span data-stu-id="94900-113">In .NET 5 and later versions, most CAS-related APIs are obsolete and produce compile-time warning `SYSLIB0003`.</span></span>


<span data-ttu-id="94900-114">이는 컴파일 시간 전용 변경입니다.</span><span class="sxs-lookup"><span data-stu-id="94900-114">This is a compile-time only change.</span></span> <span data-ttu-id="94900-115">.NET Core의 이전 버전에서 런타임 변경은 없습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-115">There is no run-time change from previous versions of .NET Core.</span></span> <span data-ttu-id="94900-116">.NET Core 2.x - 3.x에서 작업을 수행하지 않는 메서드는 .NET 5 이상에서도 런타임에 계속 작업을 수행하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-116">Methods that perform no operation in .NET Core 2.x - 3.x will continue to perform no operation at run time in .NET 5 and later.</span></span> <span data-ttu-id="94900-117">.NET Core 2.x - 3.x에서 <xref:System.PlatformNotSupportedException>을 throw하는 메서드는 .NET 5 이상에서도 런타임에 <xref:System.PlatformNotSupportedException>을 계속 throw합니다.</span><span class="sxs-lookup"><span data-stu-id="94900-117">Methods that throw <xref:System.PlatformNotSupportedException> in .NET Core 2.x - 3.x will continue to throw a <xref:System.PlatformNotSupportedException> at run time in .NET 5 and later.</span></span>

## <a name="reason-for-change"></a><span data-ttu-id="94900-118">변경 이유</span><span class="sxs-lookup"><span data-stu-id="94900-118">Reason for change</span></span>

<span data-ttu-id="94900-119">[CAS(코드 액세스 보안)](../../../../framework/misc/code-access-security.md)는 지원되지 않는 레거시 기술입니다.</span><span class="sxs-lookup"><span data-stu-id="94900-119">[Code access security (CAS)](../../../../framework/misc/code-access-security.md) is an unsupported legacy technology.</span></span> <span data-ttu-id="94900-120">CAS를 사용하도록 설정하는 인프라는 .NET Framework 2.x - 4.x에만 있지만 사용되지 않으며 서비스 또는 보안 수정을 받지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-120">The infrastructure to enable CAS exists only in .NET Framework 2.x - 4.x, but is deprecated and not receiving servicing or security fixes.</span></span>

<span data-ttu-id="94900-121">CAS가 사용되지 않기 때문에 [지원 인프라가 .NET Core](../../../porting/net-framework-tech-unavailable.md) 또는 .NET 5.0+으로 전달되지 않았습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-121">Due to CAS's deprecation, the [supporting infrastructure was not brought forward to .NET Core](../../../porting/net-framework-tech-unavailable.md) or .NET 5.0+.</span></span> <span data-ttu-id="94900-122">그러나 앱이 .NET Framework 및 .NET Core에 대해 크로스 컴파일을 수행할 수 있도록 API가 전달되었습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-122">However, the APIs were brought forward so that apps could cross-compile against .NET Framework and .NET Core.</span></span> <span data-ttu-id="94900-123">따라서 일부 CAS 관련 API가 존재하고 호출할 수 있지만 런타임에 작업을 수행하지 않는 “열기 실패” 시나리오가 됩니다.</span><span class="sxs-lookup"><span data-stu-id="94900-123">This led to "fail open" scenarios, where some CAS-related APIs exist and are callable but perform no action at run time.</span></span> <span data-ttu-id="94900-124">그러면 런타임에서 CAS 관련 특성 또는 프로그래밍 API 호출을 적용하도록 하는 구성 요소에 대한 보안 문제가 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-124">This can lead to security issues for components that expect the runtime to honor CAS-related attributes or programmatic API calls.</span></span> <span data-ttu-id="94900-125">런타임에서 이러한 특성이나 API를 사용하지 않음을 더욱 잘 전달하기 위해 .NET 5.0에서는 대부분 사용되지 않음으로 설정했습니다.</span><span class="sxs-lookup"><span data-stu-id="94900-125">To better communicate that the runtime doesn't respect these attributes or APIs, we have obsoleted the majority of them in .NET 5.0.</span></span>
